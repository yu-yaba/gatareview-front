# レビュー閲覧制限 手動切替仕様書

対象: `gatareview-front` / `gatareview-back`  
更新日: 2026-03-20

## 1. 背景

授業レビューサイトとして、通常時はレビュー投稿を促したい。  
一方で、必要な時期には管理者判断でレビュー閲覧制限を解除したい。

そのため、**管理者が手動でレビュー閲覧制限を ON / OFF できる単一スイッチ**を導入する。

## 2. 現行実装の整理

現行実装には `ReviewPeriod` があり、アクティブ期間がある場合はその期間内の投稿数、ない場合は累計投稿数でレビュー閲覧権限を判定している。  
また、管理者向けの `ReviewPeriod` API は存在するが、レビュー閲覧制限そのものを直接 ON / OFF する仕組みはない。

2026-03-20 時点で、本番DBの `review_periods` にレコードは存在しないことを確認済み。  
そのため、`ReviewPeriod` 廃止にあたって既存の期間データ移行は不要とする。

本仕様では、この `ReviewPeriod` ベースの設計は廃止対象とする。

## 3. 目的

- 管理者がレビュー閲覧制限を手動で ON / OFF できるようにする
- 影響範囲を `/lectures/[id]` の授業詳細ページ内レビューに限定する
- 現在レビューを 1 件以上保持しているログインユーザーには閲覧権限を付与する
- ルートページや最新レビューなど、他ページの公開仕様は変えない
- フロントとバックエンドで判定を統一する

## 4. 閲覧制御ルール

レビュー閲覧制限の状態は以下の2値とする。

- `OFF`
  - 授業詳細ページのレビュー本文を全ユーザーに全文公開する
- `ON`
  - 授業詳細ページのレビュー本文に閲覧制限を適用する

`ON` 時の閲覧条件は以下とする。

- ログイン済み
- 現在の投稿レビュー数が 1 件以上

判定基準は「現在の投稿レビュー数が 1 件以上」であり、期間制限は設けない。

そのため、レビューを削除して現在の投稿レビュー数が 0 件になったユーザーは、レビュー閲覧制限が `ON` の間は再び制限対象になる。

以下のユーザーは制限対象とする。

- 未ログインユーザー
- ログイン済みだが現在の投稿レビュー数が 0 件のユーザー

匿名投稿については以下とする。

- 未ログイン状態で投稿したレビューは、ユーザーの投稿レビュー数に加算されない
- 閲覧権限の判定対象は、ログインユーザーに紐付いたレビュー投稿のみとする

## 5. 影響範囲

制御対象は以下とする。

- `/lectures/[id]` の授業詳細ページに表示されるレビュー本文

制御対象外は以下とする。

- ルートページの最新レビュー
- 授業一覧、検索結果、その他の要約表示
- レビュー件数
- 平均評価
- レビュー投稿画面
- マイページの集計情報
- `ありがとう`
- `報告`
- `ブックマーク`
- 本人レビューの編集、削除

`ありがとう`、`報告`、`ブックマーク`、本人レビュー編集削除などの機能は、閲覧制限の ON / OFF に関係なく従来どおり表示する。  
このうち、`ありがとう`、`ブックマーク`、本人レビュー編集削除はログイン必須とし、`報告` は現行どおり利用可能とする。

## 6. 制限表示の挙動

レビュー閲覧制限が `ON` で、かつ閲覧条件を満たさない場合は、現行の制限表示を維持する。

- 先頭レビューは全文表示する
- 2 件目以降は本文の一部のみ表示し、続きはぼかす
- レビュー投稿導線のモーダルを表示する

レビュー閲覧制限が `OFF` の場合は、授業詳細ページ内の全レビュー本文を全文表示する。

## 7. 管理仕様

管理者が操作する設定は **1 つだけ** とする。

- `レビュー閲覧制限`
  - `ON`
  - `OFF`

`ReviewPeriod` は使用しない。  
管理UIは `ReviewPeriod` 画面ではなく、レビュー閲覧制限専用画面 `/admin/review-access` として実装する。

想定する管理画面表示項目は以下とする。

- 現在のレビュー閲覧制限状態
- `ON / OFF` の切替スイッチ
- 最終更新日時
- 最終更新者

切替時の確認ダイアログは以下とする。

- `OFF` から `ON` に変更する場合
  - タイトル: `レビュー閲覧制限を有効にしますか？`
  - 本文: `未ログインユーザーとレビュー未投稿ユーザーは、授業詳細ページのレビュー全文を閲覧できなくなります。`
- `ON` から `OFF` に変更する場合
  - タイトル: `レビュー閲覧制限を無効にしますか？`
  - 本文: `すべてのユーザーが、授業詳細ページのレビュー全文を閲覧できるようになります。`

保存完了時のメッセージは以下とする。

- `レビュー閲覧制限を有効にしました`
- `レビュー閲覧制限を無効にしました`

保存失敗時のメッセージは以下とする。

- `レビュー閲覧制限の更新に失敗しました`
- 通信失敗時は `通信エラーが発生しました。時間をおいて再度お試しください`

## 8. 管理者権限

現行実装では、管理APIにアクセスできるのは `User#admin?` が true になるユーザーのみである。  
`admin?` は `ADMIN_EMAILS` または `ADMIN_EMAIL` 環境変数に含まれるメールアドレスを持つユーザーを管理者として扱う。

本仕様でも、レビュー閲覧制限の切替は管理者のみ可能とする。

管理画面アクセス時の挙動は以下とする。

- 未ログインユーザーは `/auth/signin` へリダイレクトする
- ログイン済みだが非管理者のユーザーには `403 Forbidden` を返す

## 9. データ設計

`ReviewPeriod` は廃止し、レビュー閲覧制限専用の単一設定を保持する。

保持する値は以下とする。

- `lecture_review_restriction_enabled`
  - boolean
  - `true` で制限 ON
  - `false` で制限 OFF
- `last_updated_by_user_id`
- `updated_at`

設定の保存先は、単一レコードの `site_settings` テーブルまたは同等の永続化手段とする。

`site_settings` 未作成時とは、初回デプロイ直後や、設定レコードが存在しない状態を指す。  
この場合は安全側で `lecture_review_restriction_enabled = false` とみなし、制限 `OFF` として扱う。  
初回マイグレーションまたは seed で単一レコードを作成するが、アプリケーション側でも未作成時のフォールバックを持つ。

## 10. API 仕様

### 10.1 公開側 API

`GET /api/v1/lectures/:lecture_id/reviews` のレスポンスは、レビュー配列だけではなく、閲覧制御情報を含む形式に変更する。

このAPI変更は後方互換を持たせず、フロントエンドとバックエンドを同時リリース前提で反映する。

想定レスポンス:

```json
{
  "reviews": [],
  "access": {
    "restriction_enabled": true,
    "access_granted": false
  }
}
```

レビューが 0 件の授業でも、`reviews` は空配列で返し、`access` は必ず返す。

```json
{
  "reviews": [],
  "access": {
    "restriction_enabled": true,
    "access_granted": false
  }
}
```

`access_granted` の判定は以下とする。

- `restriction_enabled = false` の場合は常に `true`
- `restriction_enabled = true` の場合は「ログイン済みかつ現在の投稿レビュー数 1 件以上」のとき `true`

### 10.2 管理側 API

レビュー閲覧制限専用の管理 API を用意する。

- `GET /api/v1/admin/review-access`
  - 現在の制限状態を返す
- `PATCH /api/v1/admin/review-access`
  - 制限状態を更新する

更新可能な値は `lecture_review_restriction_enabled` のみとする。

## 11. フロント挙動

- 授業詳細ページは、API の `access.restriction_enabled` と `access.access_granted` を受け取る
- 授業詳細ページでレビュー本文を表示するかどうかの最終判定は `access.access_granted` のみを使う
- `localStorage` を使った権限付与やフォールバック判定は廃止する
- レビュー投稿成功時にフロントだけで権限を保存する実装は廃止する
- `reviews_count` はバックエンドが `access.access_granted` を計算するための判定材料として使う
- 制限 `OFF` のときは、バックエンドが `reviews_count` に関係なく `access.access_granted = true` を返す
- 制限 `ON` のときは、バックエンドが `reviews_count` を用いて `access.access_granted` を計算する
- レビュー取得APIが失敗した場合は、`reviews_count` などで代替判定せず、レビュー取得エラーとして表示する
- `reviews.length === 0` の授業ではロックUIを表示しない
- `reviews.length === 0` の授業では、レビュー未投稿状態の通常UIのみを表示する

## 12. 受け入れ条件

- 管理者が手動でレビュー閲覧制限を ON / OFF できる
- `OFF` のとき、全ユーザーが授業詳細ページのレビュー本文を全文閲覧できる
- `ON` のとき、未ログインユーザーと現在の投稿レビュー数 0 件のログインユーザーは制限される
- `ON` のときでも、現在の投稿レビュー数 1 件以上のログインユーザーは全文閲覧できる
- レビューを削除して現在の投稿レビュー数が 0 件になったユーザーは、再び制限対象になる
- ルートページなど授業詳細ページ以外の公開仕様は変わらない
- `ありがとう`、`ブックマーク`、本人レビュー編集削除は従来どおりログイン必須のままである

## 13. 備考

この仕様は、レビュー閲覧制御を「期間管理」ではなく「管理者による単一スイッチ」で扱うためのもの。  
実装では `ReviewPeriod` 依存を削除し、授業詳細ページのレビュー閲覧制御だけを対象に整理する。

本番DBの `review_periods` に既存レコードがないため、`ReviewPeriod` / `UserReviewPeriodCount` 削除時に考慮すべき移行対象データは存在しない。
