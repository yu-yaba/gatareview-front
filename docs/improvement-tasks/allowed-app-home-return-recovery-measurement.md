# 許可アプリ利用後のホーム復帰時間 計測メモ

最終更新日: 2026-04-23
更新理由: 計測内容を公開向けに要約し、掲載用の表現へ整えた。

## 概要

ロック中でも、電話やSMSのような一部アプリだけを一時的に使える構成を前提に、許可アプリの利用後にホーム画面へ戻ったとき、遮断用オーバーレイが再表示されるまでの時間を計測した。

この経路は通常のロック中とは少し性質が異なる。許可アプリの利用後にロック状態への戻りが遅いと、ユーザーは「もうロックに戻っているはずなのに、まだ操作できる」と感じやすいため、切り出して確認した。

## 結論

- 総遅延時間の中央値は 663 ms だった。
- 遅延の大部分は、ホーム画面に戻ったことを監視側が検知するまでの時間が占めていた。
- 表示側の描画よりも、前面アプリの監視経路を先に改善する方が効果が大きいと判断した。

## 計測の見方

ホーム画面へ戻ってからオーバーレイが再表示されるまでを、次の3点に分けて見た。

| 観測点 | 何を見たいか |
| --- | --- |
| ホーム復帰の観測 | OS上でホーム画面が前面になった事実を、監視の入口でいつ読み取れたか |
| ホーム復帰の判定 | その前面アプリ変化を受け取り、「許可アプリ利用後のホーム復帰なので遮断を戻すべきだ」といつ判断できたか |
| オーバーレイ再表示完了 | ユーザーから見て、遮断用オーバーレイがいつ再び見える状態になったか |

この3点を分けることで、遅い場所が監視側なのか、判定側なのか、表示側なのかを切り分けやすくしている。

## 計測条件

- 端末: Android Emulator
- Android バージョン: 16
- API level: 36
- build: debug
- 権限: Overlay、Usage access、Exact alarm、通知権限を有効化

## 実行手順

各回の条件をそろえるため、操作は手動ではなく adb でそろえた。

1. ロック状態と一時許可状態を初期化する。
2. 計測用のロックを開始する。
3. 電話アプリを一時的に許可する状態を準備する。
4. 電話アプリを起動する。
5. ホームボタン入力を送り、ホーム画面へ戻る。
6. ログから、ホーム復帰の観測時刻、判定時刻、オーバーレイ再表示時刻を回収する。

## 実測結果

### 集計値

| 指標 | 最小値 | 中央値 | 最大値 |
| --- | ---: | ---: | ---: |
| 総遅延時間 | 628 ms | 663 ms | 679 ms |
| 検知遅延 | 621 ms | 654 ms | 672 ms |
| 判定後の再表示時間 | 7 ms | 9 ms | 11 ms |

- 失敗は 0 回だった。
- 総遅延時間と検知遅延がほぼ同じ水準で並んでいた。
- 判定後の再表示時間はかなり短く、主な遅延要因ではなかった。

### 読み取り

今回の条件では、最も大きい遅れは、ホーム画面へ戻った事実が発生してから、監視側がそれを拾って復帰処理へ進むまでの待ち時間だった。

つまり、まず手を付けるべきなのは、オーバーレイを描画する処理そのものではなく、前面アプリを監視している経路である。

## 改善案

### 案1: polling 間隔を常に短くする

最も単純な案である。監視の polling 間隔を 750 ms から 500 ms などへ短くすれば、検知までの待ち時間は下げやすい。

- メリット
  - 実装が比較的単純である。
  - 再計測で効果を確認しやすい。
  - 今回のボトルネックに直接効く。

- デメリット
  - 通常時も polling 回数が増える。
  - CPU 使用率やバッテリー消費が悪化しやすい。
  - 必要な場面以外でも、常にコストを払うことになる。

### 案2: 一時許可セッション中だけ監視頻度を上げる

ここでいう一時許可セッション中とは、電話やSMSのショートカットを押してから、その利用が終わり、ホームへ戻ったなどと判定されて通常のロック監視へ戻るまでの短い区間を指す。

通常のロック中は今の監視頻度を維持し、この区間だけ polling 間隔を短くする案である。

- メリット
  - ユーザー体感に直結する場面へ絞って効かせられる。
  - 通常時の CPU 使用率やバッテリー消費を大きく増やさずに済む。
  - 今回の計測対象に合っている。

- デメリット
  - 状態に応じた切り替え制御が必要になり、実装はやや複雑になる。
  - セッション開始と終了の境界で、期待どおりに切り替わるかを確認する必要がある。
  - 再計測では、通常時と一時許可中の両方を比較する必要がある。

## 現時点の判断

現時点では、案2を第一候補とした。

理由は、今回の遅延が「許可アプリ利用後にホームへ戻る」という限られた場面で目立っており、その区間だけを狙って改善した方が、通常時の負荷増加を抑えやすいためである。
